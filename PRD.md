# PRD: 政府標案情資查詢系統

版本: v1.0  
日期: 2026-06-22  
定位: 主管展示型 MVP + 開發規格  
第一版範圍: 歷史標案查詢網頁，不重爬資料

## 1. 學習評估表填寫版

### 專案主題

政府標案情資查詢系統

### 製作專案的原因

我是資安產品代理商，熟悉政府標案與 SI 合作生態，但程式能力較弱。前一輪已將歷年標案資料爬下來，資料目前主要是 Excel，只有自己比較看得懂。這次希望用 AI 輔助，把既有資料做成可查詢、可展示的網頁工具，提升工作效率，也能向公司展示具體成果。

### 想達成的專案成果目標

做出一個可查詢的網頁系統。使用者輸入 SI、機關或標案關鍵字後，能列出歷年相關標案、機關、金額、得標廠商與協作廠商，並呈現採購脈絡，包含五年採購週期觀察。系統需可遠端開啟、介面清楚，能直接展示給主管看。

## 2. 專案背景

目前已具備 2020 到 2026 年的政府標案歷史資料，其中 2026 年資料暫到 3 月。資料已經過關鍵字篩選，聚焦治安、資安、網路、系統架構等和工作相關的標案。這些資料已能輔助個人工作判斷，但仍停留在 Excel 檔案形式，存在幾個問題:

1. Excel 適合個人檢視，但不適合主管或同事快速理解。
2. 資料分散在多個年度與檔案中，查詢 SI、機關或標案脈絡需要人工篩選。
3. 歷年採購週期與金額趨勢需要人工比對，展示成本高。
4. 若要向公司證明價值，需要一個清楚、可遠端展示、可互動查詢的成果。

本專案第一版不處理爬蟲問題，重點是把既有 Excel 歷史資料整理成可查詢的網頁系統。

## 3. 專案目標

### 商業目標

1. 讓業務或產品代理商能快速查出某 SI、機關或關鍵字的歷年標案紀錄。
2. 協助判斷哪些機關長期有資安或系統採購預算。
3. 協助觀察五年採購週期，例如用 2020 與 2025 對照，或用 2022 資料推估 2027 潛在機會。
4. 形成可展示給主管的成果，證明 AI 輔助開發與標案資料分析能實際改善業務工作。

### 產品目標

1. 將既有 Excel 標案資料整理為 CSV 中繼檔，並匯入 SQLite 查詢資料庫。
2. 建立一個可遠端開啟的查詢網頁。
3. 支援 SI、機關、標案名稱、關鍵字查詢。
4. 顯示歷年標案清單、金額與廠商脈絡。
5. 提供基礎統計摘要，讓主管快速看懂查詢結果的價值。

## 4. 目標使用者

### 主要使用者: 資安產品代理商業務

需要支援 SI 投標政府案，想快速知道某個機關過去和哪些 SI 合作、採購過哪些資安或系統相關標案、金額規模如何，以及是否可能進入下一輪採購週期。

### 次要使用者: 公司主管或副總

不一定熟悉原始 Excel 資料，但需要快速理解這個工具是否能幫助公司提前掌握標案機會、安排業務布局、支援原廠或 SI 合作策略。

### 潛在使用者: 其他業務或產品同仁

未來若系統穩定，可提供給其他同仁查詢特定機關、SI 或產品線相關的歷史標案。

## 5. MVP 範圍

第一版只做查詢網頁 MVP，目標是在短時間內做出可展示、可使用、可驗收的完整成果。

### 必做功能

1. 資料匯入
   - 將既有 Excel 資料先轉成 CSV 中繼檔，再匯入 SQLite。
   - 保留原始年度、機關、標案名稱、金額、得標廠商、協作廠商等欄位。
   - 若部分欄位缺失，需保留空值並在頁面清楚顯示。

2. 關鍵字查詢
   - 支援輸入 SI 名稱、機關名稱或標案關鍵字。
   - 查詢範圍至少包含機關、標案名稱、得標廠商、協作廠商。
   - 查詢結果需能跨年度搜尋。

3. 查詢結果列表
   - 顯示年度、機關、標案名稱、金額、得標廠商、協作廠商。
   - 支援依年度或金額排序。
   - 金額需格式化，方便閱讀。

4. 單筆標案詳情
   - 點選查詢結果後，可看到該標案完整欄位。
   - 若有原始資料來源或檔案來源，需顯示來源資訊。

5. 脈絡摘要
   - 顯示本次查詢結果總筆數。
   - 顯示本次查詢結果總金額。
   - 顯示年度分布。
   - 顯示主要機關或主要廠商。

6. 五年採購週期觀察
   - 以歷史資料協助觀察五年週期。
   - 第一版以「觀察」呈現，不宣稱為必然預測。
   - 可優先呈現 2020 對 2025、2022 推 2027 的案例。

7. 遠端展示
   - 系統需能在本機以外的環境開啟，例如部署到雲端或可分享網址。
   - 展示時不需要安裝 Excel 或打開程式碼。

### 不做範圍

1. 不重新爬政府採購網。
2. 不處理即時或未決標標案。
3. 不做 AI 問答助理。
4. 不做自動業務建議或自動商機評分。
5. 不做複雜登入、權限、多人協作。
6. 不做完整 BI 儀表板。
7. 不保證五年週期推估一定準確，只呈現資料支持的觀察。

## 6. 核心使用流程

### 流程一: 查詢 SI 歷年標案

1. 使用者進入查詢網頁。
2. 在搜尋框輸入 SI 名稱，例如中華電信、台灣大哥大、遠傳或其他系統整合商。
3. 系統列出該 SI 相關歷年標案。
4. 使用者查看年度、機關、標案名稱、金額、協作廠商。
5. 系統顯示該 SI 的總金額與年度分布。

### 流程二: 查詢機關採購脈絡

1. 使用者輸入機關名稱，例如警政署、法務部、地方政府或特定單位。
2. 系統列出該機關歷年相關資安或系統標案。
3. 使用者判斷該機關是否長期有固定採購預算。
4. 使用者查看過去主要合作 SI 與標案金額。

### 流程三: 查詢關鍵字與五年週期

1. 使用者輸入產品或需求關鍵字，例如治安、資安、網路設備、系統架構。
2. 系統列出相關標案。
3. 使用者檢視 2020 與 2025 是否有相似標案。
4. 系統協助標示可能具備五年週期觀察價值的結果。
5. 使用者可用 2022 年資料作為 2027 年商機布局參考。

## 7. 資料需求

### 必要欄位

1. 年度
2. 標案名稱
3. 機關名稱
4. 標案金額或決標金額
5. 得標廠商
6. 協作廠商或共同投標廠商
7. 標案類型或關鍵字分類
8. 原始資料來源或檔案來源

### 建議欄位

1. 標案案號
2. 決標日期
3. 承辦人或承辦單位資訊
4. 投標廠商清單
5. 底價或預算金額
6. 標案網址
7. 原始年度檔案名稱

### 資料清理規則

1. 金額欄位需轉成可加總的數字。
2. 年度欄位需統一格式。
3. 廠商名稱需盡量去除多餘空白與符號。
4. 機關名稱需保留原文，不強行合併，避免錯誤歸類。
5. 缺失欄位不刪除資料，改以空值或「未提供」呈現。

## 8. 頁面與介面需求

### 首頁 / 查詢頁

首頁即為主要查詢工具，不做行銷型 landing page。

頁面需包含:

1. 系統名稱: 政府標案情資查詢系統
2. 搜尋框: 可輸入 SI、機關或關鍵字
3. 篩選條件: 年度範圍、金額範圍、查詢類型
4. 查詢結果表格
5. 結果摘要區塊
6. 五年週期觀察提示區

### 查詢結果表格

每筆結果至少顯示:

1. 年度
2. 機關
3. 標案名稱
4. 金額
5. 得標廠商
6. 協作廠商
7. 查看詳情按鈕

### 詳情頁或詳情面板

點選單筆標案後顯示:

1. 完整標案資訊
2. 原始來源
3. 相關廠商
4. 相關機關
5. 可用於業務判斷的備註欄位

## 9. 技術方向

PRD 不強制指定框架，但第一版應選擇低複雜度、可快速展示的技術。

### 建議架構

1. 資料整理層
   - 使用 Python 將 Excel 轉為 CSV。
   - 再匯入 SQLite，提供穩定查詢。

2. 查詢服務層
   - 使用 Python 查詢 SQLite。
   - 提供關鍵字搜尋、排序、統計摘要。

3. 網頁介面層
   - 可使用 Streamlit、FastAPI + 前端頁面，或其他輕量方案。
   - 優先考量開發速度、展示穩定性與可維護性。

4. 部署層
   - 第一版可部署到簡單雲端環境。
   - 展示時需能透過網址開啟。

### 技術選型原則

1. 優先選簡單、穩定、容易由 AI 輔助維護的方案。
2. 避免過早導入複雜前後端分離架構。
3. 避免為了展示效果犧牲資料正確性。
4. 所有資料處理步驟需可重跑，避免手工改檔造成不可追蹤。

## 10. 驗收標準

第一版完成時，需符合以下條件:

1. 能匯入 2020 到 2026 年既有 Excel 歷史標案資料；若特定年度欄位異常，需明確記錄未匯入原因。
2. 能在網頁輸入 SI、機關或關鍵字並查到結果。
3. 查詢結果能跨年度顯示。
4. 查詢結果至少包含年度、機關、標案名稱、金額、得標廠商、協作廠商。
5. 系統能計算查詢結果總筆數與總金額。
6. 系統能顯示年度分布或讓使用者看出年度脈絡。
7. 能用一個實際案例展示五年採購週期觀察。
8. 系統能透過網址遠端開啟。
9. 主管不需要看 Excel，也能理解查詢結果與商業價值。

## 11. 展示腳本

展示給主管時，可使用以下流程:

1. 說明資料來源: 已整理 2020 到 2026 年政府標案歷史資料。
2. 說明問題: 原本資料在 Excel 中，查詢與展示成本高。
3. 打開查詢網頁。
4. 輸入一個公司熟悉的 SI 或機關。
5. 展示歷年標案清單、金額與合作廠商。
6. 展示總金額與年度分布。
7. 用一個案例說明五年採購週期觀察。
8. 說明商業用途: 可提前一年布局 SI、原廠與機關需求。

## 13. 風險與對策

### 風險一: Excel 欄位格式不一致

對策: 先挑一組代表性資料做匯入，建立欄位對照表，再逐步擴充其他年度。

### 風險二: 廠商名稱與機關名稱不一致

對策: MVP 先做原文搜尋，不急著做自動合併。後續再建立名稱對照表。

### 風險三: 五年週期被誤解為精準預測

對策: 文件與介面都使用「觀察」與「潛在機會」措辭，不使用保證性說法。

### 風險四: 功能範圍膨脹

對策: 第一版只做查詢、列表、詳情、摘要與一個週期案例。AI 問答、即時資料、商機評分全部延後。

### 風險五: 展示時資料或系統不穩

對策: 準備一組固定展示案例，並保留本機可開啟版本作為備援。

### 2026-07-02 Streamlit Community Cloud 部署狀態

1. GitHub repository 已建立：`https://github.com/paynebau/government-tender-intelligence`。
2. 本地 `main` branch 已推送到 GitHub。
3. Streamlit Community Cloud 帳號已完成註冊與登入。
4. Streamlit / GitHub 授權流程已完成。
5. 雲端部署採 `app.py` 作為入口檔，Python 依賴由 repository root 的 `requirements.txt` 安裝。
6. 雲端環境若沒有 `database/tenders.sqlite`，系統會自動從 `資料庫_CSV/*SourceData.csv` 載入資料。
7. 下一步是在 Streamlit Community Cloud 建立 app，設定：repository `paynebau/government-tender-intelligence`、branch `main`、main file path `app.py`。
8. 完成部署後，需將正式網址補回 README、PROGRESS 與 TODO，並驗證搜尋、年度篩選、智慧摘要、相似標案與異常提醒。

### 部署限制與注意事項

1. `database/tenders.sqlite` 不提交 GitHub，避免雲端部署依賴本機二進位資料庫。
2. `output/*.csv` 不提交 GitHub，避免查詢輸出汙染版本庫。
3. 目前資料來源仍是 2021-2025 年 CSV；若新增年度，需更新 `資料庫_CSV/` 並推送到 GitHub。
4. 若 repository 維持 private，Streamlit Community Cloud 必須保有 private repository 存取權限。
5. 後續若資料量變大，應評估外部資料庫或物件儲存，不宜長期把大型 CSV 直接放在 GitHub。
