# 外部資料源串接評估

更新日期：2026-07-02

## 評估結論

P5 階段先不直接串接外部 API 到正式查詢主流程。原因是目前系統已用本機 CSV 匯入 SQLite 作為可信資料源，若直接混用即時 API，會遇到欄位定義、年度範圍、查詢條件與資料更新時間不一致的問題。

建議下一步先做「離線匯入型串接」：由腳本抓取外部資料，轉成既有標準欄位，再匯入 SQLite。Streamlit app 繼續只查 SQLite，避免前端查詢速度與可重現性受 API 影響。

## 候選來源

| 來源 | 用途 | 風險 |
|---|---|---|
| 政府採購公開資料相關 API | 補新年度或缺漏標案 | 欄位格式與目前 CSV 不一定一致，需要先建立 mapping。 |
| g0v / PCC 相關資料 | 補充公開採購資料索引或社群整理資料 | 維護狀態與授權條款需逐一確認。 |
| 現有 CSV 來源 | 目前正式查詢資料源 | 需要持續確認年度完整性與編碼。 |

## 建議串接流程

1. 建立 `scripts/fetch_external_sources.py`，只負責下載與保存原始資料。
2. 建立欄位對照，輸出與 `scripts/import_to_sqlite.py` 相容的標準 CSV。
3. 對下載資料做資料品質檢查：年度、案號、機關、標案名稱、金額、得標廠商不得全空。
4. 以 staging SQLite 或暫存表比對重複案號，確認不會覆蓋既有資料。
5. 通過測試後再匯入正式 `tenders` 表。

## 暫不納入主流程的原因

- 查詢結果需要可重現，本機 SQLite 比即時 API 穩定。
- 外部 API 可能限流或欄位調整，會影響 Streamlit 使用體驗。
- 目前尚未實作登入與資料版本管理，不適合直接混入多來源即時資料。

## 下一步

先選定一個候選來源，建立小型下載樣本與欄位 mapping 測試，再決定是否納入 P6 或 v2.0。
